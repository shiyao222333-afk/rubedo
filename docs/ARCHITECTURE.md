# 凝华（Rubedo）系统架构设计 v1.0

> 一句话：把现在"所有功能塞进一个程序"的凝华，改成**边界清楚的模块化单体**——好加新 SOP、数据多了不卡、预留插头接 Nigredo / Citrinitas。暂不拆微服务、暂不脱离桌面跑。

---

## 1. 为什么改（当前痛点）

当前 v0.3.0 是单体桌面程序：界面、API、定时任务、数据层全在一个 Python 进程里，数据存在 `data/` 的 JSON 文件（靠文件锁防冲突），所有逻辑挤在 `utils.py` + `api.py`（30+ 接口混在一起），前端 11 万行 `init.js` 重度依赖 DayPilot。

五个"长不大"的隐患：

1. **数据越积越慢** — JSON 文件随记录增多变慢、文件锁并发脆弱
2. **加新业务会越来越乱** — 逻辑混在一起，改一处怕牵动别处
3. **被 DayPilot 绑死** — 前端大量依赖第三方组件（之前 19 轮才搞定其高度）
4. **自动化离不开桌面** — 定时任务跑在桌面程序内，不打开凝华就停
5. **生态接口是空的** — 蓝图写的 Nigredo→Citrinitas→凝华 闭环，凝华里没有连接代码

本次"可扩展"范围（已确认）：**易加新业务 + 数据不卡 + 接生态**；暂不做"脱离桌面跑"。

---

## 2. 目标架构：模块化单体 + 清晰边界

### 2.1 五大边界（Bounded Contexts）

把混在一起的逻辑拆成 5 个内聚模块，各管各的数据访问：

| # | 模块 | 负责 |
|---|------|------|
| 1 | **日程域** Calendar & Scheduling | 事件、重复、节假日 |
| 2 | **SOP 执行域** SOP Execution | 每条 SOP 一套流程，手动/自动切换 |
| 3 | **时间审计域** Time Audit | 计时、时薪、报表 |
| 4 | **自动化域** Automation | 定时任务、触发（APScheduler 归这里） |
| 5 | **集成适配域** Integration Adapter | 对外"插头"：Nigredo 客户端、Citrinitas 客户端 |

### 2.2 依赖方向（铁律）

```
界面 / API  →  领域服务（5 大模块）  →  数据访问层  →  存储
```

- 禁止：领域之间直接改对方内部；界面直接写文件。
- 好处：改 SOP 不动日历；加自动化不动审计；未来要拆微服务，边界已就绪。

### 2.3 存储升级（解决"数据不卡"）

- **现在**：JSON 文件 + 文件锁（易慢、易冲突）
- **目标**：换**单文件 SQLite**（同一程序、零部署、能写 SQL 查询、无文件锁冲突）
- **未来拐点**：真遇到瓶颈再换 Postgres（因存储已隔离，迁移不难）
- 详见 ADR-001

---

## 3. 对外接口（解决"接生态"）

集成适配域就是系统的"插座"：

- **Nigredo 客户端**：凝华调 Nigredo API 拉群消息 / 触发自动回复（蓝图已定：采集交给 Nigredo）
- **Citrinitas 客户端**：每单执行数据回流 → Citrinitas；未来取知识
- **形式**：每个外部系统一个 adapter，内部用统一接口定义，方便换实现 / 加模拟（mock）

生态闭环（技术版）：

```
Nigredo(采集)  --API-->  凝华[集成适配域]  -->  SOP 执行域
凝华[执行结果]  --API-->  Citrinitas(存知识)
```

> Albedo（验真假）若未来落地，置于 Nigredo 与 Citrinitas 之间，或作为独立校验服务；当前不在范围内。

## 4. 架构决策记录（ADR）

### ADR-001 存储：JSON → SQLite

- **状态**：✅ 已采纳并實施（v0.4 地基已完成，2026-07-07）
- **背景**：JSON 文件随数据增长变慢、文件锁并发脆弱
- **决策**：v0.4 起内部存储迁到单文件 SQLite；数据访问层（`modules/shared/store.py`）统一封装，领域不直接碰文件
- **后果**：+可写 SQL 查询、+并发安全（WAL + busy_timeout）；-需一次性迁移脚本（已用 migrate.py/rollback.py 闭环）
- **详细记录**：见 [`docs/ADR-001-sqlite-migration.md`](./ADR-001-sqlite-migration.md)

### ADR-002 形态：模块化单体，不拆微服务

- **状态**：提议
- **背景**：单人工具、单进程、数据量小；微服务是过度设计
- **决策**：先画清边界做模块化单体；未来要拆，边界已就绪
- **后果**：+简单可维护、+可逆；-横向扩展能力有限（当前不需要）

### ADR-003 SOP 模型：配置驱动插件

- **状态**：提议
- **背景**：加新 SOP（小红书等）不应改核心代码
- **决策**：每个 SOP = 一份配置（环节 + 手动/自动），核心读配置渲染流程
- **后果**：+易加新业务；-需设计配置格式（可复用现有 `kujiale.json` 思路）

### ADR-004 集成：外部系统一律走 Adapter

- **状态**：提议
- **背景**：领域直接调第三方会让代码耦合、难测试
- **决策**：外部系统只经集成适配域，领域不直接依赖第三方库/API
- **后果**：+可替换 / 可模拟；-多一层封装成本

## 5. 长期发展规划（分阶段，对齐现有版本路线）

| 阶段 | 版本 | 架构动作 | 对应业务 |
|------|------|---------|---------|
| **地基** | v0.4 | 划 5 边界 + 存储换 SQLite + 加集成适配域骨架（Nigredo mock） | 环节自动化 |
| **打通** | v0.5 | 真接 Nigredo + Citrinitas 回流；SOP 插件化落地（小红书可复用） | 数据回流 + AI |
| **扩展** | v0.6 | 第二业务用插件模式验证"易加新业务"；审计增强 | 小红书 SOP |
| **演进** | v1.x | 若需后台自动化 → 抽自动化域为独立 worker（此时才考虑"脱离桌面跑"） | 平台化 |

---

## 6. 风险与权衡

- **模块化单体 vs 微服务**：现在拆是过度设计；边界先清，未来好拆
- **SQLite vs Postgres**：够用到明显瓶颈；存储已隔离，迁移可控
- **DayPilot 耦合（隐患 #3）**：建议包一层"日历适配层"，换组件只改这一层，不碰业务

## 8. 已知局限与未来修订（v1.0 未覆盖，待后续迭代）

> 本节记录当前设计**有意留白**或**未来一定会踩**的点。架构已批准"先用着、未来再改"，这些就是改的方向清单。

| # | 局限 | 为什么现在是空白 | 未来修订触发点 |
|---|------|----------------|---------------|
| L1 | **前端 DayPilot 无防腐层** | 第 6 节只"建议"包一层，未定义接口；`init.js` 仍直接耦合 DayPilot | 想换日历组件 / DayPilot 升级时 → 必须先把日历操作抽成 `CalendarAdapter` |
| L2 | ~~JSON→SQLite 迁移脚本未设计~~ → **已实施** | `migrate.py`（先备份 `data/` 再灌库）+ `rollback.py`（删库恢复备份）已落地，v0.4 完成 | 无（数据有损风险已闭环） |
| L3 | **跨系统通信契约未定** | 集成适配域只留骨架，Nigredo/凝华 的"消息格式 / 传输方式 / 鉴权"没定 | v0.5 真接时 → 先定契约（建议 OpenAPI + 统一错误码），再写 adapter |
| L4 | **依赖方向未写进代码约束** | 第 2.2 节是约定，没有 lint / 架构测试强制 | 新代码易出现反向依赖（领域 import 界面）→ 未来加依赖校验 |
| L5 | **可观测性缺失** | 接生态后无跨边界追踪 / 失败告警 | v0.5 接双系统后 → 加请求 ID 透传 + 失败兜底告警 |
| L6 | **无自动化测试层** | 当前零测试；Citrinitas v1.0.1 曾因回归 bug 回退 | 加集成适配域时 → 先写契约测试保护边界，否则接外部系统必反复 |
| L7 | ~~SQLite 并发未验证~~ → **部分验证** | 已开 WAL + `busy_timeout=5000` + 单写者模型；APScheduler 定时任务并发压测待做 | 后台自动化（脱离桌面跑）时 → 重评并发模型 |
| L8 | **Albedo 位置仅占位** | 第 3 节末只说"未来可能"，无接口 | Albedo 立项时 → 明确它在闭环中的职责与接入点 |
