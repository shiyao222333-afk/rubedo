## 🔍 新发现的问题（按严重度）

### 【P2 · 设计缺口】重复任务的进度"记不进去"，但目前点不到
- **现象**：`find_event_by_id` 只去数据库里找"手动创建的任务"，但**每周自动出现的重复任务**是程序临时算出来的，不在数据库里。所以这类任务的进度点保存会失败。
- **为什么现在没事**：日历上重复任务是"只读"的（灰色、不能编辑），点击它不会显示"标记进度"按钮，所以你根本点不到——问题被前端"挡"住了。
- **风险**：这是一层脆弱的"隐式约定"。哪天你想让重复任务也能记进度，后端会悄悄失败（点了没反应、还不报错），到时候排查会费劲。
- **建议**：先记进"已知限制"，暂不修。未来要支持时，让查找函数也认得重复任务即可。

### 【P2 · 并发隐患】同一天狂点两个任务的进度，理论上可能丢一个
- **现象**：保存进度的过程是"读出来 → 改一下 → 整批写回去"。如果两个任务在同一天被**同时**修改，后写的会把先写的覆盖掉。
- **为什么现在没事**：你是单人桌面工具，几乎不可能同一毫秒点两个。上次加的 `busy_timeout` 防的是"数据库锁死报错"，但防不了这种"逻辑覆盖"。
- **建议**：当前不碰。等将来做成多人/高频并发时，改成"只更新那一条"或加版本号锁。

### 【低 · 稳健性】日志初始化顺序可以更好（一行改动）
- **现象**：`app.py` 里"启动日志"的语句，排在"导入工具模块"之后。目前工具模块在导入时不会打日志，所以没影响。
- **建议**：把启动日志挪到最前面，更稳妥（万一哪天导入时就想打日志，也不会漏）。纯防御性，非 bug。

### 【低 · 健壮性】时间记录写死"今天"
- **现象**：记一笔时间时，代码写死了"用今天日期"，不接受外部指定。正常用没问题；但跨 midnight（比如 23:59 开始记、00:01 才存）或测试场景可能写错天。
- **建议**：加个可选参数 `day`，默认今天。低优先级。

### 【低 · 优化】查找事件有点"笨"
- **现象**：按 id 找事件时，是把有事件的每一天都打开扫一遍。现在数据少无所谓；将来数据多了会慢。
- **建议**：在表里加一个"事件 id"列 + 索引，直接定位（O(1)）。纯性能优化。
