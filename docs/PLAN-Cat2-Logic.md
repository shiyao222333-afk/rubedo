## 〇、人话总览（先看懂再往下）

大类一解决了「**订单存哪**」（项目表）。大类二解决「**订单、日历事件、软件工具三者怎么连、行为怎么变**」：

- 你点开日历里一个事件 → 程序看它绑了哪个**订单** → 显示**那个订单**的 SOP 步骤和进度（而不是这张事件自己的）。
- 你点的「开始计时 / 标记完成」按钮 → 计时记进**统一记录表**，并推进**订单**的进度。
- SOP 步骤和工具长啥样 → 由 `kujiale.yaml` 文件说了算（改文件、点重载就变）。

一句话：**进度归订单，计时归统一表，面板跟着订单+SOP文件走。**

## 一、研究结论（竞品 + 参考模型）

### 1.1 git 竞品盘点

| 项目 | 类型 | 和我们像不像 | 借鉴点 |
|------|------|------------|--------|
| **Kimai**（PHP，开源） | 计时器，含 客户→项目→活动 三层 | ⭐⭐⭐ 数据模型几乎一致 | L3/L7 的「项目→步骤→计时」归属 |
| **DRYTRIX/TimeTracker** | 多客户项目计时 + 开发票 | ⭐⭐ 计费/收入视角 | time-audit（酷家乐独有）的「计时→收入」思路 |
| **ActivityWatch** | 自动计时（不用手动点） | ⭐ 自动采集思路 | 未来 Nigredo 接单可自动起停计时 |
| **yaml-workflow / flowengine** | Python YAML 工作流引擎 | ⭐⭐⭐ 步骤/流程/状态/恢复 | L2/L10 的 SOP-as-代码 + 工具执行引擎 |
| **Process Street / Tallyfy / Trainual** | SaaS SOP 清单 | ⭐⭐ 只有步骤、无计时无订单 | 步骤状态 UI 参考 |
| **awesome-selfhosted 计时类** | 一堆自托管计时器 | ⭐ 通用计时 | 无新意 |

**结论性发现（重要）**：git 上有「计时器」、有「SOP SaaS」、有「工作流引擎」，**但没有一个开源工具把「SOP步骤清单 + 每步计时 + 客户订单绑定 + SOP独有小工具」绑在一起**。我们的组合是一个空白 niche —— 可以放心走自己的设计，同时大量借 Kimai 的归属模型和 yaml-workflow 的执行模型。

### 1.2 关键参考：Kimai 的「三层归属」模型

Kimai 的核心关系（我们直接对应）：

```
客户(Customer)
  └─ 项目(Project)        ← 我们的 projects 表（一个酷家乐订单）
       └─ 活动(Activity)  ← 我们的 SOP 步骤(step_id)
            └─ 计时记录(Timesheet)  ← 我们的 timelog 表（带 project_id + step_id）
```

借鉴点：
- 计时记录**永远挂在「项目+活动」上**，不在「某次工作会话」上 → 对应我们 L3/L6「计时写 timelog 带 project_id+step_id」。
- 一个项目可跨多天、多会话累计 → 对应我们 L7「订单累计计时聚合」。
- 活动(步骤)是项目的属性，不是会话的属性 → 对应我们 L5「进度归订单」。

### 1.3 关键参考：yaml-workflow 的执行模型（给 L10）

yaml-workflow 的做法：YAML 定义 `steps`（每个 step 一个 `task`=程序）+ `flows`（执行顺序）+ 状态 + 失败可 `--resume` 从断点续跑。
借鉴到我们的「工具三模块执行引擎(L10)」：
- 每个工具 = 一个「任务」：读输入(`input`) → 跑程序(`program`) → 写输出(`output`)。
- 工具的 `input`/`output` 可绑**不同**步骤（输入绑步骤A、输出绑步骤B）→ 工具自动把两步串起来。
- 我们不做「流程编排引擎」那么重，只做「按 SOP 声明的 tools 列表逐个解析执行」——参考它的 task 模型，但砍掉 flows/condition 那些复杂部分（防架构宇航员）。

## 三、关键设计决策（带研究依据，需你拍板）

### D-C2-1：L2 的 SOP 文件校验 —— 手写轻量校验，不引第三方库
- **选项A（推荐）**：手写一个 `validate_sop(yaml_dict)` 检查必填键（id/name/stages；每 stage 有 id/name/steps；每 step 有 id/name），出错给中文红字 + 用上次好版本兜底。
- **选项B**：引 pydantic/jsonschema 做严格校验。
- **理由**：SOP 文件结构简单，引库是过度设计（违反「无架构宇航员」原则）；手写校验 + 友好报错已满足 G4。零新依赖。

### D-C2-2：L8 状态机 —— 手写转移表，不引 transitions 库
- **选项A（推荐）**：一个 `STATUS_TRANSITIONS = {pending: {active}, active: {review, on_hold, cancelled}, ...}` 字典，转移前查表，非法转移报错。
- **选项B**：引 `transitions` 库。
- **理由**：只有 9 个状态、转移规则固定，字典 10 行搞定，引库反而重。等状态涨到 20+ 再考虑。

### D-C2-3：L5 双轨 —— 保留，但因旧数据已清可极简
- 旧数据全清（你确认的），所以没有「旧进度迁移」负担。
- 保留双轨只是：有 `project_id` 走订单进度，没有就走事件自己的 `sop_current_step`。代码量很小，且更稳（防以后又有未绑事件）。**推荐保留**。

### D-C2-4：L6 计时写入 —— 必须改上一轮的路径
- ⚠️ 上一轮加的计时写进了**事件 JSON**（`sop_step_timings`），但大类一已定调「统一 timelog + 进度归订单」。这俩冲突，是数据分裂（T3 反复修的毛病）。
- **本任务强制**：timer 结束 → 写 `timelog`（带 project_id/step_id/occurrence_id）+ 更新 `projects.step_data`（黑板）。事件 JSON 不再存计时。
- 这是「修自己之前埋的雷」，必须做。

### D-C2-5：L10 工具引擎 —— 先做 timer，框架按 DESIGN 文档的注册表
- 沿用 `DESIGN-SOP-Tools-Model.md` 的 `TOOL_REGISTRY` 注册表 + 每个工具 `input/program/output` 三模块。
- 本轮只落地 `timer`（通用）：input 无 / program 计时 / output 写当前步骤的 `duration_sec` 到黑板 + timelog。
- time-audit（酷家乐独有）仍**不做**，只留框架。
- 参考 yaml-workflow 的 task 模型，但**不做 flows/condition 编排**（避免过度设计）。

## 四、依赖关系 & 执行顺序

```
L1(清类型) ─┐
L2(加载器) ─┼─→ L3(绑定解析) ─→ L4(面板按项目) ─→ L5(双轨) ─→ L6(计时写timelog) ─→ L7(聚合)
L8(状态机) ─┘                                              │
                                                          L9(接单标记) ─→ (依赖 L3/L4)
L10(工具引擎) ── 独立，但建议放最后和大类三 UI 一起调
```

**推荐顺序（每批做完你试用）**：
1. **批A（小、独立）**：L1 清类型 → L8 状态机 → L2 加载器+校验
2. **批B（逻辑核心）**：L3 绑定解析 → L4 面板按项目 → L5 双轨 → L6 计时写 timelog → L7 聚合
3. **批C（语义+引擎）**：L9 接单标记 → L10 工具引擎(timer)
