## 二、事件 / 项目 / 工具 三者关系（用真代码 + 具体例子讲清）

### 2.1 三个东西各是什么（查代码得来）

- **事件（Event）= 日历上的"时间块"**
  - 代码位置：`events` 表（SQLite），每天一张。
  - 它长这样：`{ id, text, start, end, kind, description, reminder, exec_mode, status, locked, schedule_id(重复事件才有), project_id(规划中) }`
  - 一句话：事件就是"某天某段时间我在干嘛"的一块记录。

- **项目（Project / 订单）= 一个酷家乐客户订单**
  - 代码位置：**目前不存在**，代码里 grep "project" 零匹配。这是我这版新引入的概念。
  - 它应该长这样：`{ id, name(客户/订单名), client, sop_id(用哪个 SOP 模板), income(订单金额), status(生命周期), sop_current_step(做到第几步), due_date(交期), notes(备注), created_at, updated_at }`
  - 一句话：项目是"进度 + 计时 + 收入 + 生命周期"的**家**。一个订单的所有信息都归它，是**一等公民**，不是事件上的一个字段。

- **工具（Tool）= 凝华这个软件本身（含底部 SOP 操作面板）**
  - 就是你正在用的这个程序。它的"底部面板"当你点一个事件时，会画出该项目 SOP 的当前步骤、给你"开始计时 / 标记完成"按钮。
  - 一句话：工具是**承载事件和项目、并让你推进步骤的那层界面**。

### 2.2 三者怎么连（核心关系，用例子）

```
        ┌─────────────────────────────────────────────┐
        │  工具（凝华软件）                              │
        │   • 提供日历（放事件）                         │
        │   • 提供底部面板（点事件→显示项目SOP步骤）      │
        │   • 提供项目面板（管理订单生命周期）           │
        └───────────────┬───────────────────┬──────────┘
                        │ 创建/查看          │ 点事件→显示
                        ▼                   ▼
                 ┌──────────────┐    ┌──────────────────┐
                 │  事件(时间块) │    │   项目(订单)       │
                 │ work/提醒/.. │──绑定──>│ client/收入/状态 │
                 │ project_id──>│    │ 当前步骤/每步耗时  │
                 └──────────────┘    └──────────────────┘
                                          ↑
                                  引用 SOP 模板(固定文件)
```

**具体例子（你最在意的"多项目并行 + 跨天"）：**

> 今天接了 3 个酷家乐订单：订单A、订单B、订单C。
> - **3 个项目**：A / B / C 各建一个，各自记客户名、收入、状态、当前步骤。互不干扰 ✅
> - 订单A 你**周一**下午做 1 小时、**周二**上午做 2 小时：
>   - 周一建一个事件（时间块），`project_id = A`
>   - 周二建另一个事件，`project_id = A`
>   - **进度和计时都存在项目 A 身上**，不在事件上 → 跨天不裂开 ✅
> - 你点**任意**一张绑 A 的事件 → 底部面板都显示**项目 A** 的 SOP 步骤（做到第几步、每步花多久），而不是那张事件自己的。

**这就是你直觉对的地方：工具该绑"项目"（订单），不是绑"某张便利贴（事件）"。** 事件是项目的一次"工作会话"，项目才是进度的归属。

### 2.3 事件种类清理（修正我上版的笔误）

现在代码里 `kind` 取值：`sop / tool / external / marker / reminder`。问题：
- `sop` 把"类型"和"绑定"混了 —— 它其实意思是"这张事件绑了个 SOP 项目"，但该用 `project_id` 表达，不该是一种类型。
- `tool` 也容易误导 —— "工具"是软件本身，不是一种时间块。

**建议改成（已删掉 holiday，且不再用 sop/tool 当类型）：**

| 轴 | 含义 | 取值 |
|----|------|------|
| 时间块类型 `kind` | 这块时间是什么性质 | `work`（干活，可绑项目）/ `reminder`（提醒）/ `external`（外部事件）/ `marker`（标记日） |
| 项目绑定 `project_id` | 这块时间属于哪个订单 | 可空；有值 = 在做某个 SOP 订单 |

→ `kind=sop` 删掉，SOP 关联通过 `project_id → 项目.sop_id` 表达。类型与业务绑定不再打架。

### 2.4 重复事件怎么和项目绑（微调：按天绑定 + 接单时还没项目）

重复事件（如"每天接单""每天站会"）在日历里每天生成一条 occurrence（当天实例）。**绑定分两层：**

- **模板级 `project_id`（可选）**：适用于"每次都是同一个项目"的重复事件。例如"每日运营复盘"→ 固定绑到"运营"项目，模板上写一次，每天生成的实例自动继承。
- **实例级 `project_id`（通过 `occurrence_overrides`，默认空）**：适用于"接单"类重复 —— 你每天点开那天的接单实例时，订单还没接，所以先不绑。等你当天真的接了单，再"新建项目 / 选已有项目"绑到这一天的实例上。

**关键修正（你的洞察）：接单重复事件在生成的那一刻没有固定项目，项目是在"接单动作"发生时才产生的。**

  - 接单类重复模板本身**不预设** `project_id`。
  - 每天那一条 occurrence 才是"绑定单位"，不是模板。
  - 你点开某天的接单实例 → 面板显示「尚未绑定项目」→ 你接单后填客户/收入/建项目 → 这一天的实例就绑上了，进度和计时都归这个项目。
  - 同一项目可能跨多天、绑多天的接单实例（如周一接的单周三还在做）→ 都指向同一个 `project_id`。

→ 这样既保留"重复事件在日历里重复"的便利，又让每个订单（项目）独立、可跨多天累计，且不要求接单时预先知道项目。

**解析优先级**：`occurrence_overrides.project_id` 优先于 `schedules.project_id`；都为空 = 未绑定。

### 2.5 项目要更细致的管理方法

项目不是事件上的一个字段，而是**一等公民**：独立表（`projects`）+ 独立管理面板。

**项目字段（比上版更细）：**
```
id, name(订单/客户名), client(联系人), sop_id(用哪个SOP模板),
income(订单金额), status(生命周期), sop_current_step(做到第几步),
due_date(交期, 可选), notes(备注, 可选), created_at, updated_at
```

**`status` 生命周期（让管理更细致）：**
```
pending 待接单 → active 进行中 → review 待验收 → done 已完成 → paid 已收款
（可回退，例如验收不通过退回 active）
```

**管理面板（"项目"页 / 侧栏列表）：**
  - 列出全部项目，可按 `status` 筛选。
  - 每行显示：名称 / 客户 / 金额 / 状态 / 当前步骤 / 累计计时 / 绑定事件数。
  - 操作：新建项目、改状态、改金额、看时间明细、把某个事件/实例绑到它。

**绑定入口（呼应 2.4）：**
  - 点开"未绑定"的接单实例 → 面板给「新建项目」或「选已有项目」→ 绑定这一天的实例。
  - 之后点这张（或同一项目其他天的）实例 → 都显示该项目 SOP 步骤与计时。

"更细致"体现在：项目有生命周期状态、有金额、有交期、有独立管理列表，而不是散落在各张事件上。

## 三、SOP 环节 = 固定模板文件（单真相源，改了面板自动变）

用户要求："SOP 环节需要一个固定的模板方便修改和读取" + 上一版我推荐"数据库+编辑界面(F1)"是错的，应改为**文件模板**。本版改为：

**决策：SOP 定义以 YAML 模板文件为唯一真相源，数据库只当可选缓存。**

### 3.1 模板文件长这样（`sops/kujiale.yaml`，用真实数据结构）

```yaml
id: kujiale
name: 酷家乐接单 SOP
version: 3
stages:
  - id: 1
    name: 接单
    steps:
      - id: 1.1
        name: 监控群消息
        mode: auto          # auto=将来可自动化 / manual=现在手动
        est_min: 0          # 预计耗时（分钟），时间审计用
        desc: 客服在接单群发订单需求，Nigredo 监控并提取订单摘要
        # provider/input/output 是给"将来自动化"留的，现在可空
      - id: 1.2
        name: 提取订单信息
        mode: manual
        est_min: 3
        desc: ...
  - id: 2
    name: 私聊接单
    steps: [ ... ]
  # …… 共 5 阶段 32 步（结构已存在于数据库，照搬到文件即可）
```

- 这就是"固定的模板"：**人读得懂、任意编辑器能改、程序启动时读进来**。
- 每个 SOP 一个文件（`sops/kujiale.yaml`、`sops/xxx.yaml`），加新 SOP = 加个文件。

### 3.2 "改了工具栏自动跟着变"怎么实现

- 底部面板**每次点事件都重新读最新模板文件**（不缓存旧的那份）。
- 你改完 `kujiale.yaml` 后，面板加一个 **「重新加载 SOP」按钮**（或重启程序）→ 立刻生效。
- 数据库 `sops` 表可以**删掉**，或保留为"启动时的快照缓存"。本版推荐：**直接读文件，去掉 DB 这层**，少一层同步 bug（这正是上一版 F1 的坑）。

### 3.3 为什么用文件而不是数据库界面（回答"回答有问题"）

- 你要的是"方便修改和读取" —— 文件用任意编辑器（含手机）都能改，数据库界面得专门开发且只在软件里能改。
- 文件是纯文本，能进 git 版本管理，改错了能回退、能 diff 看出改了哪步。
- 代价只是"改完点一下重新加载"，相比 F1 要开发整套编辑 UI，简单得多，也更符合你"手动化优先、自动化以后慢慢来"的节奏。

## 五、数据模型落点（不写代码，只说存哪）

1. **新增"项目"表 `projects`**：`id, name, client, sop_id, income, status, sop_current_step, due_date, notes, created_at, updated_at`。
2. **事件表 `events` 加 `project_id`**（可空）：时间块挂到项目。
3. **重复模板表 `schedules` 加 `project_id`**（可空，可选）：适用于"每次同项目"的重复（如运营复盘）。
4. **重复实例表 `occurrence_overrides` 加 `project_id`**（可空）：每天那一条实例的绑定，接单类在此绑。
   - 解析优先级：`occurrence_overrides.project_id` 优先于 `schedules.project_id`；都为空 = 未绑定。
5. **每步耗时日志表 `sop_step_logs`**（审计用）：`project_id, event_id, occurrence_id, step_index, step_id, started_at, completed_at, duration_sec, mode`。
   - 手动/自动完成在这里长得一样（只 `mode` 不同）—— 为将来自动化留好口子。
6. **SOP 定义改为 YAML 文件**（`sops/*.yaml`），DB `sops` 表降级为可选缓存或删除。
